# Git Algorithms and Internals

Git's efficiency and reliability stem from deterministic algorithms and structures designed for "snapshot-based" distributed version control. Unlike older systems (CVS, SVN) that used file-locking and deltas, Git treats the entire state of the repository as a series of snapshots.

## The Diff Algorithm: Longest Common Subsequence (LCS)
Calculating the difference between two files is a computationally expensive task. Git utilizes the **Longest Common Subsequence (LCS)** algorithm to identify the minimum set of changes (insertions/deletions) required to transform one file into another.
- **Complexity**: Naive implementations are exponential. Git uses optimized heuristics (like the Myers algorithm) that approach $O(M \times N)$ time complexity.
- **Hunks and Chunks**: A "patch" is composed of contiguous blocks of modified lines called **hunks** (or chunks). Each hunk includes context lines to ensure the patch applies correctly even if surrounding lines have changed.

## Git Objects: Commits and Trees
- **Snapshot Paradigm**: Instead of storing the difference between version A and B, Git stores the content of version B as a new **Blob** object. 
- **The Merkle Layer**: As discussed in [[Merkle_Trees_and_DAGs|Merkle Trees and DAGs]], Git uses cryptographic hashes (SHA-1) of these blobs to build **Trees** and **Commits**, ensuring data integrity across the entire history.

## The "Remendo" (Patch) Workflow
Historically, developers exchanged modifications using `.patch` files generated by the `diff` utility and applied via the `patch` utility. Git automates this process but maintains the underlying philosophy: a commit is essentially a metadata-wrapper around a tree state, but it can be exported and applied as a standard Unix patch.

## [[Local_Git_Resilience|Local Git Resilience]] and Custom Remotes

Git is "distributed" by design, meaning every clone is a full backup of the state and history. Local resilience strategies leverage this without relying on centralized hubs (GitHub/GitLab).

### 1. Bare Clones as Backup Remotes
A **Bare Repository** (`git init --bare` or `git clone --bare`) contains only the Git administrative data and object database, without a working directory for editing.
- **Resilience Tactic**: Maintaining a bare clone on an external high-durability medium (Pendrive, External HD, or NAS).
- **Workflow**: 
  - `git clone --bare /path/to/project /media/ext-hdd/project.git`
  - `git remote add backup /media/ext-hdd/project.git`
  - `git push backup main`

### 2. Lightweight Git Ecosystem: Serving via HTTP
For scenarios where a centralized server isn't available, Git can be served over standard HTTP with minimal overhead:
- **Serving**: `python3 -m http.server 8000` (within the `.git` directory of a bare repo).
- **Prerequisite**: Use `git update-server-info` to generate the necessary index files for the "dumb" HTTP protocol.
- **Cloning**: `git clone http://<ip-address>:8000/ project-clone`

This pattern ensures that as long as the bitstream is accessible via standard network protocols, the repository can be fully reconstructed.
