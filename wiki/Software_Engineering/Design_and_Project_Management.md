# Design and Project Management

Software development is a duality between **Computer Science** (the mathematical foundations) and **Software Engineering** (the craftsmanship and industrial production).

## Science vs. Engineering

Building a bridge requires physical constants and proven civil engineering. Building software requires internalizing these differences:

- **Computer Science**: Foundational theory (Algorithms, Data Structures, Compilers, OS Internals). Knowing *how* the computer works at a fundamental level.
- **Software Engineering**: The application of science to solve practical problems (Clean Code, Agile, Design Patterns, CI/CD). Knowing *how* to build, maintain, and scale products.

## Emerging Design

A common failure in project management is **BDUF (Big Design Up Front)**—trying to specify every detail before writing a single line of code.

### The Emerging Approach
Instead of a rigid blueprint, design should "emerge" as the system is built:
- **Refactoring**: Constant iteration to improve code structure as requirements become clearer.
- **Design Patterns**: Strategic use of patterns (Strategy, Factory, etc.) only when they solve an actual problem, rather than over-engineering from day one.
- **Continuous Integration**: Validating that the "evolving" design doesn't break existing functionality.

## Foundational Literature
- *Design Patterns* (GoF): For language-specific implementation structures.
- *Clean Code* (Robert C. Martin): For human-readability standards.
- *Smalltalk Best Practice Patterns* (Kent Beck): Foundational thinking for object-oriented design.
- *Growing Object-Oriented Software, Guided by Tests*: For TDD and emergent architecture.
